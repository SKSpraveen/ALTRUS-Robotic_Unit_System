## 1. Project Ecosystem Overview

ALTRUS is a **modular, fault‑tolerant assistive robotics software ecosystem** composed of:

* **ALTRUS Core Middleware** – Intent‑driven, fault‑aware robotic middleware
* **ML Model V2 CLI** – Sensor‑fusion ML project scaffolding tool
* **Wristband Sensor Fusion Project** – Generated anomaly & activity detection runtime
* **Robo Emotion Framework** – Multimodal (face + voice) emotion detection with LLM interaction

Each component can operate independently, but together they demonstrate a **full research‑grade assistive robotic software stack**.

## 2. ALTRUS ML Model V2 CLI

### Overview

A Python CLI that scaffolds **standalone wristband sensor‑fusion ML projects** for:

* Activity classification
* Physiological anomaly detection

The CLI collects sensor, anomaly, and activity selections, then generates a ready‑to‑edit project with configuration, simulation stubs, and ML model hooks.

### Installation

```bash
pip install -e .
```

### Usage

```bash
altrus init
```

You will be prompted to select:

* Sensors
* Anomalies
* Activities
* Target environments
* Output location

A YAML configuration file is generated so selections can be updated later without rebuilding the project.

### Running a Generated Project

```bash
cd <project-name>
altrus run
```

### Training Workspace

Use `training_workspace/` to generate demo ML models.

```bash
python training_workspace/train_models.py
```

Generated `.pkl` files should be copied into the project `models/` directory.

## 3. Wristband Sensor Fusion Project

### Overview

This project is generated by the **ALTRUS ML Model V2 CLI** and provides a runnable runtime for sensor ingestion, simulation, and anomaly detection.

Configuration file:

```
config/wristband_config.yaml
```

### Quick Start

Run live UDP scanner:

```bash
altrus run
```

Run live TCP scanner:

```bash
altrus run --protocol tcp --port 5055
```

Run UDP simulation test:

```bash
python -m tests.run_simulation
```

### Manual Anomaly Simulations

Run these in a separate terminal while the scanner is active:

```bash
python -m simulations.run_tachycardia
python -m simulations.run_bradycardia
python -m simulations.run_fever
python -m simulations.run_heart_attack
python -m simulations.run_cardiac_arrest
```

### Models

Default models:

```
models/activity_model.pkl
models/anomaly_model.pkl
```

Replace them with trained models as needed.

## 4. Robo Emotion Framework

### Overview

A **multimodal emotion detection framework** combining:

* Face emotion recognition (CNN)
* Voice emotion recognition (MFCC‑based NN)
* Multimodal fusion
* LLM‑powered conversational assistant (Ollama)
* Speech recognition (Whisper) and synthesis (pyttsx3)

### Key Features

* Real‑time webcam emotion detection
* Continuous audio emotion analysis
* Weighted multimodal fusion
* Emotion‑aware LLM responses

### Project Structure

```
robo_emotion_framework_test/
├── src/
│   ├── config.py
│   ├── multimodal_ollama_assistant.py
│   ├── assistant.py
│   ├── asr/
│   ├── tts/
│   ├── llm/
│   └── models/
├── models/
│   ├── face_emotion_detection_model.h5
│   └── voice_model_best.h5
├── notebooks/
├── framework_cli.py
└── requirements.txt
```

### Installation & Setup

**Python:** 3.10 – 3.13

```bash
python -m venv .venv
source .venv/bin/activate
pip install -r requirements.txt
```

### Ollama Setup

```bash
ollama serve
ollama pull gemma3:1b
```

Configure in `src/config.py` if needed.

### Run the Application

```bash
python -m src.multimodal_ollama_assistant
```

### CLI Project Generator

Install CLI:

```bash
pip install -e .
```

Create a new project:

```bash
python framework_cli.py init
```

The CLI scaffolds a full runnable emotion detection project.

### Configuration

Adjust audio, fusion, and label settings in:

```
src/config.py
```

### Model Training

* Voice: `notebooks/voice_modal_train.ipynb`
* Face: `notebooks/face_emotion_train.ipynb`

Expected dataset structure:

```
audio_dataset/
├── happy/
└── sad/
```

### Troubleshooting

* Ensure Ollama is running
* Verify model files exist
* Check webcam/microphone permissions
* Use Python 3.10–3.13

### Performance Metrics

**Voice Model**

* Validation Accuracy: ~91.5%
* Test F1‑Score: ~0.85

**System**

* ~30 FPS video
* 2–5s LLM response latency

## 5. ALTRUS Core Middleware

### Project Overview

**ALTRUS Core** is the middleware backbone of the ALTRUS ecosystem.

It provides:

* Intent‑driven execution
* Fault tolerance & recovery
* Policy‑based routing
* Immutable audit logging

It **does not implement physical robots**, only the supervisory software architecture.

### Problem Statement

Traditional assistive robots suffer from:

* Tight coupling
* Fragile pipelines
* No intent lifecycle
* Poor fault isolation
* No trustworthy audit trail

ALTRUS Core addresses all of these.

### High‑Level Architecture

Intent Sources → Intent Engine → Routing Policies → Module Registry → Fault Engine → Blockchain Ledger

### Core Components

#### Module Registry

* Single source of truth
* Persistent storage (`~/.altrus/registry.json`)
* Lifecycle & health tracking

States:

* REGISTERED, ACTIVE, DEGRADED, FAILED, OFFLINE

Health:

* HEALTHY, UNSTABLE, CRITICAL, UNKNOWN

#### Health Monitoring

* Heartbeat‑based evaluation
* Automatic escalation
* Ledger logging

#### Fault Engine

* Detects failures
* Performs recovery
* Supports extensible strategies

#### Intent Engine

Lifecycle:

```
CREATED → VALIDATED → EXECUTING
          ↓
       PREEMPTED / REJECTED
```

Features:

* Priority arbitration
* Capability‑based routing
* Fault‑triggered preemption

#### Policy‑Based Routing

* YAML‑driven
* Runtime configurable
* Health‑aware selection

#### Blockchain‑Backed Ledger

* Append‑only blocks
* Immutable event history
* Full auditability

#### Observability

Prometheus metrics:

```
http://localhost:8000/metrics
```

Metrics include:

* Module counts
* Intent counts
* Fault & recovery counters

### CLI Demonstration

Module operations:

```bash
altrus modules add
altrus modules activate
altrus modules heartbeat
altrus modules list
```

Intent operations:

```bash
altrus intents inject
altrus intents reconcile
altrus intents list
```

Ledger inspection:

```bash
altrus ledger
```

### Completed Scenarios

✔ Module lifecycle management
✔ Heartbeat‑based failure detection
✔ Intent injection & execution
✔ Policy‑based routing
✔ Fault‑triggered preemption
✔ Persistent state
✔ Immutable audit logging
✔ Prometheus metrics

### Current Limitations

* No physical robot integration
* No real hardware drivers
* Mock recovery strategies
* No UI/dashboard

### Planned Work (Phase 2)

* Advanced intent lifecycle (TTL, SLA)
* Event‑driven architecture
* Multiple recovery strategies
* Visualization dashboards
* ROS / gRPC integrations

### Key Contributions

* Fault‑aware intent execution
* Policy‑driven routing engine
* Blockchain‑backed audit trail
* CLI‑first middleware testing
